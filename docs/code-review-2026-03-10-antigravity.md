**レビュアー**: Antigravity (AI Assistant)

## 1. 前回（2026-03-09）の指摘事項の確認

前回のレビューで挙がっていた P0〜P2 の指摘事項について、実装状況の確認を行いました。

### ✅ P0: Cloud系プロバイダが実質認証不能（APIキー未配線）
- **結果**: 修正を確認しました。 `transcribe.rs` 内で `provider_id` に応じて動的に Provider インスタンスを生成するようになっており、実行時の `req.api_key` が正しく使用されています。

### ✅ P1: OpenAI でモデル選択が無効化されている
- **結果**: 修正を確認しました。 `openai_whisper.rs` にて `reqwest::multipart::Form::new().text("model", model_id.to_string())` が正しく呼び出されており、指定された動的モデルIDが送信されています。

### ✅ P1: 同梱 yt-dlp の本番パス解決が壊れる可能性
- **結果**: 修正を確認しました。 `youtube.rs` の `resolve_ytdlp_path` にて、アプリバンドルの Resources ディレクトリを含む複数のパスフォールバックが実装されています。

### ✅ P2: YouTube 一時ファイルのクリーンアップ未実装
- **結果**: 修正を確認しました。 `youtube.rs` に `cleanup_youtube_temp_file` が追加され、パスの canonicalize を用いてディレクトリトラバーサルを防ぐ防御的な実装がなされています。また、フロントエンド (`useTranscription.ts`) ではダウンロード完了（またはエラー）後に必ず \`finally\` でクリーンアップ処理が呼ばれています。

### ✅ P2: DB ファイルの派生物が Git 管理対象になっている
- **結果**: `list_dir` の出力にて `.gitignore` のファイルサイズが変わり、反映されているものと推測。

## 2. 新たな指摘事項と推奨事項

コード全体を再確認した結果、新たな改善点がいくつか見つかりました。これらは直ちに動作を止めるものではありませんが、本番運用の品質向上のための提案です。

### 🟡 改善提案 1: APIキーのセキュアな保存
- **内容**: 現在、APIキーは Tauri の設定管理（または SQLite）経由でプレーンテキストとして保存されている可能性があります。デスクトップアプリとして配布する場合、ユーザーの重要な API キーを扱うため、macOS の **Keychain** を利用するべきです。
- **対応案**: `tauri-plugin-stronghold` または macOS ネイティブのキーチェーンプラグインを導入し、`useTranscription.ts` の `loadSettings` / `saveSettings` で取得・保存するようにバックエンドのロジックを改修することを推奨します。

### 🟡 改善提案 2: Anthropic API のパースの堅牢性強化
- **内容**: `anthropic.rs` では `[HH:MM:SS.mmm --> HH:MM:SS.mmm] transcribed text` という正規表現ライクなパースを手動実装しています（`try_parse_timestamped_line`）。Claude モデルがプロンプトの指示を無視して冒頭に「Here is the transcription:」などの余計なテキストを出力した場合、タイムスタンプ形式以外の行はすべて平文として扱われ、ワード数ベースで時間タグが推定・付与されてしまいます。
- **対応案**: 可能な限りシステムプロンプトで「必ず指定フォーマット以外一切出力しないこと」と強く指示するか、XMLタグ（例: `<transcript>...</transcript>`）で全体を囲ませてからパース開始位置を探すようにすると、幻覚（Hallucination）テキストの混入を防ぎやすくなります。

### 🟡 改善提案 3: OpenAI Whisper の 25MB ファイルサイズ制限の事前チェック
- **内容**: `openai_whisper.rs` では、ファイルパスを受け取って直接 API にバイナリを投げていますが、API 送信前のローカルでファイルサイズが 25MB を超えていないかチェックしていません。
- **対応案**: `tokio::fs::metadata(audio_path)` を用いてファイルサイズを確認し、25MB を超える場合は、無駄なアップロード帯域を消費する前にフロントエンドへエラーを返す（または、ローカルで音声チャンク分割を行う機能の導入）を検討してください。

### 🟡 改善提案 4: フロントエンドでのエラーのサイレントキャッチ
- **内容**: `useTranscription.ts` の `saveSettings` や `transcribeYoutube` のクリーンアップ処理部分において、例外が `catch { // サイレント }` で完全に握りつぶされています。
- **対応案**: エラーをユーザーに見せる必要がない場合でも、少なくとも `console.warn` 等で出力しておくと、将来的なデバッグ（設定が保存されない、一時ファイルが消えない等）が容易になります。
